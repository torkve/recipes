# Семейная кулинарная книга

Простой быстрый веб-сервис для хранения семейных рецептов: открытый просмотр,
редактирование для авторизованных членов семьи, полнотекстовый поиск,
динамические категории и двухсторонняя синхронизация с заметками Apple iCloud.

- Бэкенд на Go, без CGO (чистый Go-драйвер SQLite).
- Фронтенд — простой статический JavaScript, раздаётся сервером (без npm/react).
- Хранение в одном файле SQLite + каталог загруженных изображений; бэкап — архив каталога `data/`.
- Развёртывание на одном хосте через Docker.

## Сборка и запуск (локально, офлайн)

В этом окружении компилятор Go не в `PATH`. Его путь определяется так:

```sh
GO=$(ya tool go --print-path)
```

Сборка/тесты используют только локальный кэш модулей (`~/go/pkg`), поэтому
прокси указывает на файловый кэш:

```sh
export GOPROXY="file://$HOME/go/pkg/mod/cache/download"
export GOSUMDB=off
export GOFLAGS=-mod=mod

GO=$(ya tool go --print-path)
"$GO" build ./...
"$GO" test ./...
```

Запуск:

```sh
ADMIN_USERNAME=admin ADMIN_PASSWORD=secret "$GO" run ./cmd/recipes
# открыть http://localhost:8080
```

|----------------------------------|----------------------------------|--------------------------------------------------------------------------------|

| `RECIPES_DATA_DIR`               | `./data`                         | Каталог БД, загрузок и сессий (бэкапить его)                                   |
| `RECIPES_SITE_NAME`              | `Семейная кулинарная книга`      | Название сайта в шапке                                                         |
| `RECIPES_SECURE_COOKIES`         | `false`                          | Помечать cookie как Secure (только HTTPS)                                      |
| `ADMIN_USERNAME`                 | —                                | Логин первого администратора (bootstrap)                                       |
| `ADMIN_PASSWORD`                 | —                                | Пароль первого администратора (bootstrap)                                      |

## Развёртывание (Docker)

Система не распределённая — один контейнер на одном хосте. Данные хранятся в
именованном томе `recipes-data` (смонтирован в `/data`).

```sh
cp .env.example .env        # задать ADMIN_USERNAME / ADMIN_PASSWORD и пр.
docker compose build
docker compose up -d
# сайт на http://<хост>:8080 (порт настраивается через RECIPES_PORT)
```

Бинарник собирается без CGO (чистый Go) — образ статический и самодостаточный
(шаблоны и статика встроены в бинарник через `go:embed`). За TLS поставьте
обратный прокси и выставьте `RECIPES_SECURE_COOKIES=true` (см. ниже).

## Обратный прокси (nginx) и HTTPS

Сервис рассчитан на работу в **корне отдельного хоста** (например,
`recipes.example.com`), а не на подпути вида `example.com/recipes/`: приложение
генерирует абсолютные URL от корня (`/static`, `/admin`, `/uploads`, AJAX-запросы)
и не умеет работать из подкаталога. Для подпути потребовалась бы поддержка
базового пути в самом приложении (шаблоны, маршруты, JS, cookie) — это отдельная
доработка.

**1. Привяжите контейнер к localhost**, чтобы наружу его видел только nginx — в
`.env`:

```sh
RECIPES_PORT=127.0.0.1:8080   # compose отдаёт "${RECIPES_PORT}:8080"
```

```sh
docker compose up -d
```

**2. Конфиг nginx** (редирект HTTP→HTTPS + проксирование на контейнер). Важно
сохранять заголовок `Host` (от него зависят проверки CSRF Origin/Referer и
привязка cookie) и поднять лимит размера тела — вставляемые/загружаемые картинки
рецептов превышают стандартный 1 МБ nginx:

```nginx
server {
    listen 80;
    server_name recipes.example.com;
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl http2;
    server_name recipes.example.com;

    # TLS-сертификаты (см. certbot ниже)
    ssl_certificate     /etc/letsencrypt/live/recipes.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/recipes.example.com/privkey.pem;

    client_max_body_size 25m;   # загрузка изображений рецептов

    location / {
        proxy_pass http://127.0.0.1:8080;
        proxy_http_version 1.1;
        proxy_set_header Host              $host;
        proxy_set_header X-Real-IP         $remote_addr;
        proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_read_timeout 60s;
    }
}
```

```sh
sudo nginx -t && sudo systemctl reload nginx
```

**3. TLS через Let's Encrypt** (certbot сам пропишет пути к сертификатам и
редирект):

```sh
sudo certbot --nginx -d recipes.example.com
```

**4. Включите Secure-cookie** — раз сайт теперь по HTTPS, в `.env`:

```sh
RECIPES_SECURE_COOKIES=true
```

```sh
docker compose up -d
```

> Полностью в Docker: можно поднять nginx в той же сети compose и проксировать на
> `http://recipes:8080;` — тогда публиковать порт на хост не нужно. У приложения
> нет WebSocket, заголовки `Upgrade`/`Connection` не требуются.

## Синхронизация с iCloud

Apple не предоставляет публичного API для Заметок, поэтому клиент использует
приватные веб-сервисы iCloud (CloudKit), как это делают `icloud.js` /
`iCloud-API`. Функция выключена по умолчанию (`RECIPES_ICLOUD_ENABLED=false`) и
поставляется «тёмной»: маршруты `/admin/sync` и фоновый воркер не активны, пока
её явно не включат.

После включения: войдите в админку → **iCloud**, укажите Apple ID и пароль,
введите код двухфакторной аутентификации, выберите папку Заметок. Подпапки
становятся категориями; заметки — рецептами (заголовок → название, чеклисты →
ингредиенты, тело → шаги). Входящие изменения подтягиваются автоматически,
исходящие отправляются по кнопке; неоднозначности разрешаются вручную в разделе
конфликтов. Живая синхронизация требует реальных учётных данных и проверяется
только онлайн.

Вход использует протокол SRP-6a (как у iCloud в браузере). Точная байтовая
схема SRP подтверждается только онлайн, а Apple ограничивает частые попытки
входа (повторный запрос подряд возвращает 503). Поэтому при привязке делается
одна попытка с вариантом `RECIPES_ICLOUD_SRP_VARIANT` (по умолчанию 1). Если в
логах `signin/complete` вернул 401 — поставьте следующее значение (0..5),
перезапустите сервис и попробуйте снова. В логах строка `SRP variant N
accepted` означает успех.

## Поиск (полнотекстовый + семантический)

Полнотекстовый поиск (SQLite FTS5) работает всегда. Дополнительно можно включить
**семантический поиск по смыслу/синонимам** (например, запрос `блины` находит и
`блинчики`, и `оладьи`) — он использует самостоятельно размещаемую модель
эмбеддингов и объединяется с полнотекстовым результатом (Reciprocal Rank Fusion).
По умолчанию выключен; без него поиск работает как раньше.

Включение:

1. В `.env` раскомментируйте `RECIPES_EMBED_URL=http://embedder:80`.
2. Запустите вместе с контейнером модели:

   ```sh
   docker compose --profile semantic up -d
   ```

Модель (`RECIPES_EMBED_MODEL`, по умолчанию `intfloat/multilingual-e5-small`)
скачивается при первом старте в том `embedder-models`. Векторный индекс хранится
в БД, но является производным и **перестраивается автоматически** из рецептов
фоновым воркером (бэкап не нужен). Если модель недоступна, поиск автоматически
откатывается к полнотекстовому. При смене модели меняйте `RECIPES_EMBED_MODEL`
**и** `RECIPES_EMBED_DIM` вместе (это запускает переиндексацию). Размерность
обязана совпадать с моделью:

| `RECIPES_EMBED_MODEL`              | `RECIPES_EMBED_DIM` |
|------------------------------------|---------------------|
| `intfloat/multilingual-e5-small`   | `384` (по умолчанию)|
| `intfloat/multilingual-e5-base`    | `768`               |
| `intfloat/multilingual-e5-large`   | `1024`              |
| `BAAI/bge-m3`                      | `1024`              |

Если семантический поиск выдаёт слишком много нерелевантного — поднимите порог
`RECIPES_EMBED_MIN_SCORE` (например, до `0.84`–`0.88`) и перезапустите сервис;
если, наоборот, находит слишком мало — снизьте. Это главный регулятор
релевантности (косинусная близость, у e5 значения обычно высокие). Смена модели
(см. выше) улучшает само ранжирование, но порог нужен в любом случае.

## Бэкап

Все данные — в каталоге `data/` (или томе `recipes-data`): файл SQLite,
загруженные изображения, сессии и ключи. Для бэкапа остановите сервис и
заархивируйте каталог/том:

```sh
docker compose stop
docker run --rm -v recipes-data:/data -v "$PWD":/backup alpine \
  tar czf /backup/recipes-backup.tgz -C /data .
docker compose start
```
